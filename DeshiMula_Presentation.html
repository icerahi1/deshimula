DeshiMula Database System

An Anonymous Employer Review and Rating Platform for Bangladesh's IT Sector

Developed by: Rahi Chowdhury

Course: Database Practice

Institution: Vilnius College / University of Applied Sciences

Project Background & Context

In Bangladesh's fast-growing software industry, professionals frequently experience problems like payroll delays, opaque performance reviews, and unpaid overtime.

Because the market lacks open forums and workers fear retaliation or blacklisting, these issues remain hidden. Job seekers are forced to accept offers blindly, often walking into toxic workplaces.

DeshiMula (representing local transparency) solves this by providing a secure, anonymous web platform where employees can rate local employers, share honest reviews, and comment on workplace discussions without revealing their identity.

Domain Entities & Attributes

  • Core Core-Data: Companies details, employee ratings, text review contents, and user comment threads.
  • Integrity: Moderation logs, upvotes/downvotes tracking, and report flags.
  • Aesthetics & UX: Modern glassmorphism UI built with Tailwind CSS, utilizing automatic routing and filters.

Project Objectives & Core Tasks

1. Schema Architecture

Design a 15-table relational database model resolving all multi-table relationships, constraints, and data types.

2. Normalization to 3NF

Deconstruct all logical entities to Third Normal Form to eliminate data redundancy and prevent insert or delete anomalies.

3. Implementation

Build a responsive Next.js application that integrates with SQLite, using triggers to automate statistics calculations.

Academic & Practice Requirements Fulfilling

Each of the 15 database tables must contain over 100 entries. I implemented three separate data seeding mechanisms (forms, SQL, and Python bulk migration) and crafted 30 custom SQL queries to query and validate the dataset.

Relational ER Model & 3NF Rules

We designed the schema around 15 entities to completely decouple reviews, metadata, user actions, and system auditing:

  • users & audit_logs
  • companies & reviews
  • review_sentiments
  • comments & votes
  • tags & review_tags
  • industries & company_industries
  • company_locations
  • location_junction
  • reports & notifications

Applying 3NF Normalization

  • First Normal Form: Discarded comma-separated strings. Multi-valued elements (like tags) are mapped to atomic rows in a junction table.
  • Second Normal Form: Removed partial dependencies. Lookup tables (like industries) exist independently from companies.
  • Third Normal Form: Removed transitive dependencies. We link companies to their location options using IDs, preventing duplicate city-text fields.

Database Seeding Strategy

To test database search performance and integrity rules under load, every table was seeded with at least 100 records (up to 1,000+ relationships):

  • 1. App UI Forms: Users can sign up, create companies, and write reviews to test real-world INSERT operations.
  • 2. Raw SQL Seeds: DML queries insert core lookup records, defaults, and system parameters.
  • 3. Python Migration Script: A custom script parsed raw.csv containing nearly 2,000 real-world reviews, formatting slugs and resolving relationships.

Actual Table Seed Counts

Analytical SQL Queries

I crafted 30 detailed SQL queries to validate the schema, extract complex logs, and compile analytics. Below are two representative examples:

Query 15: Concatenating Locations and Industries

SELECT c.name, 
       GROUP_CONCAT(DISTINCT i.name) AS industries, 
       GROUP_CONCAT(DISTINCT l.name) AS locations
FROM companies c
LEFT JOIN company_industries ci ON ci.company_id = c.id
LEFT JOIN industries i ON i.id = ci.industry_id
LEFT JOIN company_location_junction clj ON clj.company_id = c.id
LEFT JOIN company_locations l ON l.id = clj.location_id
GROUP BY c.id LIMIT 10;

Query 26: Companies with Clean Records

SELECT c.name FROM companies c 
WHERE EXISTS (
  SELECT 1 FROM reviews r 
  JOIN review_sentiments s ON s.id = r.sentiment_id
  WHERE r.company_id = c.id AND s.name = 'Positive'
) AND NOT EXISTS (
  SELECT 1 FROM reports p 
  JOIN reviews r ON r.id = p.review_id
  WHERE r.company_id = c.id AND p.status = 'pending'
) LIMIT 10;

Web Application Architecture & Deploy

I developed a production-ready application integrated with our SQLite backend:

  • Framework: Next.js 16.2.6 (React 18) with secure Server Actions and dynamic routing.
  • Database Client: better-sqlite3 for rapid synchronous execution of raw SQL.
  • Vercel Serverless Workaround: Since serverless functions have a read-only filesystem, we copy the seeded database to the writeable /tmp directory on startup. This allows users to write reviews and comment on the live site.

Security & Moderation

  • Authentication: Cookie-based secure sessions.
  • Audit Logs: Database automatically logs actions like deletes and logins for administrator audits.
  • Database Triggers: SQLite triggers manage and recalculate aggregate scores, preventing race conditions.

Project Conclusions & Takeaways

Key Deliverables

  • 3NF Schema: Designed a robust, circular-reference-free layout of 15 relational tables.
  • Database Triggers: Deployed trigger routines to ensure statistical accuracy in the frontend.
  • Production Hosting: Deployed successfully to Vercel with a native binary configuration and writeable filesystem workaround.
  • Performance: Indexed key foreign keys (e.g. review company IDs) to keep search speeds fast.

Lessons & Next Steps

  • Serverless Hosting: Running SQLite in serverless containers is possible but requires copying files to memory-mapped filesystems (like /tmp).
  • Future Scope: Migrate to a globally distributed SQLite client like Turso/libsql to support persistent write operations.
  • AI integration: Add an automated classifier to analyze review sentiment during insert.
