DATABASE PRACTICE PROJECT

DeshiMula Database System

An Anonymous Employee Review & Rating Platform for the IT and Corporate Sector in Bangladesh

Developed by: Rahi Chowdhury

Course: Database Practice

Institution: Vilnius College / University of Applied Sciences

Project Background & Context

The job market in Bangladesh lacks a transparent forum where employees can share honest feedback about their employers.

Issues such as salary delays, lack of increments, toxic work culture, nepotism, and micromanagement are common but rarely discussed publicly due to fear of retaliation.

DeshiMula provides a secure, anonymous web platform for employees to post constructive, detailed reviews and rate companies, fostering accountability and helping job seekers make informed career decisions.

Target Application Domain

  • Entities: Companies, Reviews, Comments, Users, Locations, and Industries.
  • Interactivity: Upvotes/downvotes, reporting content, nested user discussions, and admin moderation.
  • AI Utility: Real data seeding (extracting semantic context from a large dataset of employee comments).

Objectives & Core Tasks

1. Schema Design

Construct a multi-table relational schema covering all requirements, resolving relationships, junctions, and constraints.

2. Normalization

Apply database normalization rules up to Third Normal Form (3NF) to minimize data redundancy and prevent anomalies.

3. Implementation

Develop a robust web application using Node.js, Next.js, and better-sqlite3 with server actions and a beautiful Tailwind CSS interface.

Key Practice Requirements

At least 15 relational entities in the ER model. Seeding at least 100 records in each table using manual entries, raw SQL, and bulk AI transfers. Crafting and executing 30 analytical SQL queries.

Relational ER Model (15 Entities)

The system utilizes exactly 15 entities to map the full lifecycle of employee reviews, comments, locations, and system auditing:

  • users (accounts)
  • companies (employers)
  • reviews (employee posts)
  • review_sentiments
  • comments (discussion)
  • votes (helpful ratings)
  • tags (keywords)
  • review_tags (junction)
  • industries (sectors)
  • company_industries
  • company_locations
  • company_location_junction
  • reports (flagged posts)
  • notifications
  • audit_logs (system logs)

Normalization Highlights (3NF)

  • 1NF: All attributes contain atomic values; composite arrays (like tags) are split into rows.
  • 2NF: Junction tables resolve many-to-many relationships (e.g. review tags), ensuring full dependency on primary keys.
  • 3NF: Transitive dependencies are removed. Sentiment labels are placed in `review_sentiments` and locations in `company_locations` instead of duplicates.

Data Seeding & Import Strategy

The practice required populating all 15 tables with at least 100 records each. We achieved this through three pathways:

  • 1. Visual UI: Web forms for users to sign up, submit reviews, and post comments.
  • 2. SQL Insertion: DDL and DML scripts containing core seed structures for roles and industries.
  • 3. Bulk Data Transfer: A Python migration script parsing raw.csv containing 1,999 records.
  • 4. AI Data Generation: Random generation functions inside Python seeder simulating votes, reports, notifications, audit logs, and sentiments.

Seed Counts Verified

  • companies: 293 entities
  • users: 120 user accounts
  • reviews: 408 detailed reviews
  • tags & locations: 100 each
  • review_tags: 1,019 relations
  • comments & votes: 150 comments, 300 votes
  • logs & notifications: 150 each

Analytical SQL Queries

We developed and executed 30 comprehensive queries to test database integrity, join structures, and subqueries. Key queries include:

-- Query 15: Companies with locations & industries
SELECT c.name, 
       GROUP_CONCAT(DISTINCT i.name) AS industries, 
       GROUP_CONCAT(DISTINCT l.name) AS locations
FROM companies c
LEFT JOIN company_industries ci 
     ON ci.company_id = c.id
LEFT JOIN industries i 
     ON i.id = ci.industry_id
LEFT JOIN company_location_junction clj 
     ON clj.company_id = c.id
LEFT JOIN company_locations l 
     ON l.id = clj.location_id
GROUP BY c.id LIMIT 10;
-- Query 26: Companies with Positive Reviews
-- and No Pending Reports (Correlated Subquery)
SELECT c.name FROM companies c 
WHERE EXISTS (
  SELECT 1 FROM reviews r 
  JOIN review_sentiments s ON s.id = r.sentiment_id
  WHERE r.company_id = c.id AND s.name = 'Positive'
) AND NOT EXISTS (
  SELECT 1 FROM reports p 
  JOIN reviews r ON r.id = p.review_id
  WHERE r.company_id = c.id AND p.status = 'pending'
) LIMIT 10;

Web Application Architecture

DeshiMula is developed as a production-grade Web App:

  • Framework: Next.js 15 (React 18) with App Router.
  • Database Engine: SQLite integrated with better-sqlite3 for ultra-fast, zero-network-latency server queries.
  • Styling: Tailwind CSS 3 for a modern, dark glassmorphism layout, featuring responsive cards, modals, and input fields.
  • State & Auth: Cookie-based secure session tokens, password hashing, and role-based access checks (user vs. admin).

Core Modules

  • lib/db.ts: Singleton database connection manager wrapper.
  • lib/actions.ts: Secure Server Actions for form submissions (register, login, submitReview, addComment, upvote/downvote).
  • components/review-card.tsx: Reusable cards representing review statistics and sentiment badges.

Conclusions & Recommendations

Key Conclusions

  • Implemented a highly normalized relational database schema (15 entities, 3NF).
  • Seeded the database with 100+ high-quality records per table using a unified CSV migration script.
  • Successfully executed 30 complex analytical queries, validating correctness of the indexes, joins, and triggers.
  • Created a user-friendly, responsive interface that operates cleanly in any educational computer lab.

Future Recommendations

  • AI Sentiment Auto-Tagging: Integrate HuggingFace or OpenAI to scan incoming review text and auto-assign sentiment and tag categories.
  • Data Pagination: Implement cursor-based pagination for reviews to scale with larger database dumps.
  • Notifications Engine: Switch from polling to WebSockets for real-time moderator flag alerts.
